# Changes Made on 2026-03-08

## Summary

Today we completed a UX-driven refactor of the client booking experience: moved primary booking to root (`/`), added auth modal flow (login/signup) for guests, removed redundant user entry routes, enriched admin appointment rows with user/professional names, and introduced new user-facing navigation with history/profile pages plus authenticated profile APIs.

---

## 1) Admin Appointments UX: Show Names Instead of Raw IDs

- Extended appointment payload enrichment to include:
  - `user_full_name`
  - `professional_full_name`
- Admin appointments table now renders full names with ID as secondary text.
- Files:
  - `src/services/appointments.ts`
  - `app/admin/appointments-management/page.tsx`

---

## 2) Password Utility Cleanup

- Removed legacy/unused `passwordHashing` helper from `lib/utils.ts`.
- Kept active auth hashing flow unchanged in `src/services/users.ts`:
  - signup: `hashPassword` (salt + `scryptSync`)
  - login: `verifyPassword` + `timingSafeEqual`
- File:
  - `lib/utils.ts`

---

## 3) New Auth Components for Reuse

- Added reusable user auth components:
  - `components/users/login-form.tsx`
  - `components/users/signup-form.tsx`

---

## 4) Root (`/`) Became the Main Public Booking Entry

- Replaced placeholder root page with full booking experience.
- Guests can browse slots; booking action requires auth.
- If guest attempts booking:
  - auth modal opens
  - pending slot is retained
  - after login, booking resumes for that slot
- Modal supports view switching:
  - Login
  - Sign up
- Root also shows logged user�s next upcoming appointment.
- File:
  - `app/page.tsx`

---

## 5) Route Protection Redirect Target Updated

- Proxy redirects for protected routes now target root (`/`) instead of old user entry pages.
- Preserved JWT validation logic and role checks.
- File:
  - `proxy.ts`
