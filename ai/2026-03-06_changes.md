# Changes Made on 2026-03-06

## Summary

Today we moved from basic role pages to route-based admin/professional panels, replaced custom session tokens with JWT (`jose`), added secure cron-triggered slot generation endpoints, expanded slot generation/re-generation logic, and standardized date handling strategy around UTC in storage with local-time display in frontend.

---

## 1) Authentication: Custom Token -> JWT Library

- Installed `jose` and replaced custom HMAC token implementation.
- Updated session module:
  - `src/lib/auth-session.ts`
  - `createSessionToken` now signs standard JWT (`HS256`).
  - `verifySessionToken` now verifies JWT via `jwtVerify`.
- Updated async callsites:
  - `app/api/users/login/route.ts`
  - `app/api/users/route.ts`
  - `app/api/appointments/route.ts`
  - `proxy.ts` (now async to await verification)

---

## 2) Calendar Booking UI + Slots API Shape

- Reworked user booking UX into calendar/day-selection flow:
  - Added reusable component: `components/slot-calendar.tsx`
  - Integrated in `app/user/page.tsx`
- Changed slots API to return calendar-ready rows (not only free rows):
  - `app/api/slots/route.ts`
  - `src/services/slots.ts` -> `getCalendarSlots()`
- Calendar response now includes:
  - `availability: "available" | "occupied"`
  - `appointmentId`
- Availability is computed from slot state + appointment existence.

---

## 3) Slot Generation + Re-generation Endpoints

- Added generation endpoints:
  - `POST /api/slots/generate`
  - `POST /api/slots/regenerate`
- Files:
  - `app/api/slots/generate/route.ts`
  - `app/api/slots/regenerate/route.ts`
- Removed `days` query param override from these endpoints (uses service default/env only).
- Added operational logs in generate route:
  - start, unauthorized, success summary, service error, unexpected error

---

## 4) Cron Security

- Added bearer-token validation for cron endpoints:
  - `src/lib/cron-auth.ts`
- Endpoints now require:
  - `Authorization: Bearer <CRON_SECRET>`
- If invalid/missing token:
  - returns `401 Unauthorized`

## 9) Services: `updated_at` Writes and Status Sync

- `src/services/appointments.ts`:
  - Writes `updated_at` when creating/updating/assigning/cancelling appointments.
  - Includes `updated_at` in select payloads.
  - Booking availability accepts both slot states:
    - `available` and `scheduled`
- Added slot-state sync on appointment changes:
  - On booking -> slot state set to `scheduled`
  - On status changes -> slot state updated accordingly
  - Also updates `slots.updated_at`
- `src/services/slots.ts`:
  - Slot writes include `updated_at`.
  - Slot reads include `updated_at`.

## 10) Date/Timezone Standardization Implemented Today

- Final strategy applied:
  - DB data stored/treated in UTC
  - Frontend displays converted to local timezone
- Frontend local display conversion for slot date/time:
  - `components/slot-calendar.tsx`
  - `app/admin/appointments-management/page.tsx`
  - `app/professional/appointments/page.tsx`
  - `app/professional/history/page.tsx`
- Generation logic adjusted to use template values as UTC-ready inputs (no runtime local->UTC offset conversion in generator).

## 11) Database Schema/Contract Updates (Reflected in App)

The following DB changes are now assumed/used by code:

1. `appointment_status` enum
- Active values include:
  - `scheduled`
  - `completed`
  - `cancelled`
  - `no_show`
  - `available` (newly used for slot state)

2. `slots` table
- Uses `state` with `available` as valid value.
- Uses `updated_at` column (read/write in services).

3. `appointments` table
- Uses `updated_at` column (read/write in services and UI sort/filter behavior).

4. Existing prior change still active
- `appointments.professional_id` assignment flow remains in place.
